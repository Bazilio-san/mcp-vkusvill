# Развёртывание MCP-сервера на удалённый сервер

Скилл разворачивает уже существующий проект MCP-сервера (на базе `fa-mcp-sdk`) на удалённый Linux-сервер,
запускает его как системную службу и настраивает ежеминутное автообновление из git-репозитория. Публикация
наружу выполняется через обратный прокси Caddy с автоматическим выпуском TLS-сертификата Let's Encrypt.

Все вспомогательные скрипты лежат в `${CLAUDE_SKILL_DIR}/scripts/` и запускаются через `node`.

## Главное правило: вся конкретика — в deploy.yaml

В этом файле SKILL.md **намеренно нет** ни адресов серверов, ни доменов, ни путей, ни ключей. Любые реквизиты
подключения скрипты читают из файла `deploy.yaml` в корне проекта. Шаблон со всеми пояснениями находится рядом:
`${CLAUDE_SKILL_DIR}/deploy.example.yaml`.

Если `deploy.yaml` отсутствует — попросите пользователя скопировать шаблон в корень проекта и заполнить его,
либо помогите заполнить, задавая вопросы обычной прозой (адрес сервера, пользователь, путь к ключу, домен).
Файл `deploy.yaml` содержит секреты и **обязан** быть в `.gitignore` (строка `/deploy.yaml`).

## Что задаётся в deploy.yaml

Часть значений можно не указывать — скрипт выводит их сам:

- `ssh.host`, `ssh.user`, `ssh.key` — доступ по SSH (обязательно).
- `paths.projectDir` — каталог проекта на сервере (обязательно). `paths.nodeProjectsDir` можно не задавать —
  берётся родительский каталог `projectDir`.
- `git.sshUrl` — можно оставить пустым: формируется из `package.json` → `repository.url`, где https-адрес
  преобразуется в SSH-адрес вида `git@host:owner/repo.git` (клонирование на сервере идёт по SSH).
- `git.branch` — по умолчанию `master`.
- `node.version` — можно не задавать: берётся из `.envrc` (`nvm use <версия>`).
- `service.name` — по умолчанию `package.json` → `name`. `service.port` — по умолчанию
  `config/default.yaml` → `webServer.port`.
- `domain.name`, `domain.upstream` — если домен не задан, публикация через Caddy пропускается,
  сервис остаётся доступен только локально на сервере.
- `deploy.email` — адрес для писем об автообновлении (пусто — письма не шлются).

## Требования к серверу

- Доступ по SSH с ключом из `ssh.key`.
- Установленный `nvm` с нужной версией Node.js (та, что в `node.version` / `.envrc`).
- Доступ сервера к git-репозиторию по SSH (deploy-ключ или доступ учётной записи).
- Для публикации наружу — установленный и работающий Caddy с файлом `/etc/caddy/Caddyfile`
  и каталогом логов `/opt/log/caddy/` (владелец `caddy:caddy`). DNS A-запись домена должна указывать на сервер.
- `direnv` желателен, но не обязателен (для интерактивной работы в каталоге проекта).

## Шаг 1 — Проверить deploy.yaml и показать план

Запустите оркестратор в режиме предпросмотра — он прочитает `deploy.yaml`, выведет итоговые параметры
(включая автоматически выведенные) и покажет bash-скрипт, который выполнится на сервере, **ничего не меняя**:

```
node ${CLAUDE_SKILL_DIR}/scripts/deploy.cjs --dry-run
```

Сверьте с пользователем адрес сервера, путь проекта, репозиторий, ветку, домен и порт. Если чего-то не хватает,
скрипт сообщит, какие поля `deploy.yaml` не заполнены.

## Шаг 2 — Развернуть

```
node ${CLAUDE_SKILL_DIR}/scripts/deploy.cjs
```

Скрипт идемпотентен — повторный запуск не ломает уже развёрнутый сервис. По SSH на сервере выполняется
следующая последовательность:

1. **Клонирование.** Создаётся `paths.nodeProjectsDir`, репозиторий клонируется по SSH в `paths.projectDir`.
   Если каталог уже содержит git-репозиторий — шаг пропускается.
2. **Файлы окружения.** Записывается `.env` (`SERVICE_NAME`, `NODE_ENV=production`, `DEBUG=config-info`) и
   `deploy/config.yml` (`branch`, `email`). Существующий `.env` не перезаписывается без флага `--force-env`.
3. **Права и direnv.** Ставится бит исполнения на `deploy/srv.cjs` и `update.cjs`; если есть `direnv`,
   выполняется `direnv allow .`.
4. **Сборка.** Подгружается `.envrc` (`nvm use`), затем `yarn install --frozen-lockfile` и `yarn cb`.
5. **Секреты.** Если `config/local.yaml` ещё нет, генерируется новый: случайный `encryptKey` (UUID) для
   `webServer.auth.jwtToken` и случайный `permanentServerToken`. Без них сервер не стартует, если в
   `adminPanel.authType` включён `jwtToken`. Существующий файл не трогается — секреты остаются стабильными.
6. **Служба.** `deploy/srv.cjs install` создаёт systemd-юнит `/etc/systemd/system/<service>.service`
   (`EnvironmentFile=.env`, `ExecStart=<nvm-node> dist/src/start.js`, `Restart=always`) и запускает его.
   Затем служба перезапускается, чтобы подхватить свежую сборку, и проверяется её активность.
7. **Домен.** Если задан `domain.name` и на сервере есть Caddy: заранее создаётся лог-файл
   `/opt/log/caddy/<домен>.log` с владельцем `caddy:caddy` (иначе Caddy не сможет его открыть),
   в конец `/etc/caddy/Caddyfile` добавляется блок обратного прокси на `domain.upstream`, выполняется
   `caddy validate` и `systemctl reload caddy`. Сертификат Let's Encrypt выпускается автоматически за 5–20 секунд.
8. **Автообновление.** В crontab root добавляется ежеминутный запуск `update.cjs` нужной версией Node.js.
9. **Алиас для `restart`.** Обязательный шаг. Если на сервере есть утилита `restart` (обёртка над
   `systemctl restart <service>`), регистрируется короткий алиас сервиса командой `restart add <алиас> <service>`.
   Алиас — это первые буквы слов в имени сервиса (`mcp-vkusvill` → `mv`). Если алиас уже занят другим сервисом,
   к нему добавляется цифра (`mv2`, `mv3`, …). После этого сервис перезапускают вручную командой `restart <алиас>`.
   Список текущих алиасов — `restart ls`.

## Шаг 3 — Проверить работоспособность

```
node ${CLAUDE_SKILL_DIR}/scripts/check-server.cjs
```

Скрипт по SSH проверяет: активность службы, прослушивание порта, ответ `/health` локально и по HTTPS через домен,
корректный ответ MCP на метод `initialize`, наличие cron-задания. Все проверки должны быть зелёными
(коды HTTP 200 и непустой `serverInfo`).

## Как устроено автообновление

Каждую минуту cron запускает `update.cjs` в каталоге проекта. Скрипт:

1. Откатывает любые локальные изменения (`git reset --hard`, `git clean -fd`).
2. Делает `git fetch` и сравнивает текущий коммит с вершиной отслеживаемой ветки.
3. Если есть новые коммиты — переустанавливает зависимости (`yarn install --frozen-lockfile`), пересобирает
   (`yarn cb`) и перезапускает службу (systemd или PM2 — определяется автоматически).
4. При заданном `deploy.email` отправляет письмо об успехе или ошибке.

Важно: `git clean -fd` **не удаляет** файлы, попавшие под `.gitignore`. Поэтому созданные при развёртывании
`.env`, `deploy/config.yml` и `config/local.yaml` (с секретами) переживают автообновления — их достаточно
создать один раз.

## Ручное управление на сервере

```
systemctl -l status <service>            # статус
journalctl -o cat -xefu <service>        # живой просмотр логов
systemctl restart <service>              # перезапуск
restart <алиас>                          # перезапуск через короткий алиас (см. restart ls)
node deploy/srv.cjs reinstall            # полная переустановка службы (см. deploy/srv.sh.readme.md)
```

## Опции скрипта развёртывания

- `--dry-run` — показать параметры и удалённый скрипт, ничего не выполняя.
- `--force-env` — перезаписать `.env` на сервере (иначе существующий сохраняется).
- `--config <path>` — взять конфигурацию из другого файла вместо `deploy.yaml`.

## Диагностика

**`Не найден файл deploy.yaml`** — скопируйте `${CLAUDE_SKILL_DIR}/deploy.example.yaml` в корень проекта
под именем `deploy.yaml` и заполните.

**`git.sshUrl не выведен`** — в `package.json` нет `repository.url`, либо он в нестандартном формате.
Задайте `git.sshUrl` в `deploy.yaml` явно (например, `git@github.com:owner/repo.git`).

**Служба не стартует, в журнале `encryptKey is missing or too short`** — не создан `config/local.yaml`
с секретами. Это делает скрипт развёртывания; проверьте, что файл появился на сервере и служба перезапущена.

**Caddy reload завершается ошибкой `permission denied` на лог-файле** — лог-файл нового сайта должен
существовать с владельцем `caddy:caddy` ещё до перезагрузки. Скрипт создаёт его сам; при ручной правке
выполните `install -o caddy -g caddy -m 0644 /dev/null /opt/log/caddy/<домен>.log` перед `systemctl reload caddy`.

**Сертификат не выпускается** — проверьте, что DNS A-запись домена указывает на сервер (`dig +short <домен>`)
и порт 80 доступен снаружи (Caddy использует HTTP/TLS-ALPN challenge).

**Автообновление не срабатывает** — проверьте `crontab -l` и путь к бинарю Node.js в cron-строке; он должен
указывать на установленную через nvm версию (`~/.nvm/versions/node/v<версия>/bin/node`).
